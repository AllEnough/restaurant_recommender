# 專題報告大綱：智慧飲食決策系統

## 一、研究動機

- 外食資訊分散，使用者必須自行比較價格、距離、評分與評論。
- 冰箱食材有保存期限，單純食譜搜尋無法回答「現在應該先煮什麼」。
- 現有推薦常只給結果，缺少推薦原因、風險與資料來源。

## 二、專題目標

1. 建立同時服務外食與內食族的飲食決策系統。
2. 讓推薦結果先呈現，降低操作與閱讀成本。
3. 以多條件與評論風險產生可解釋的餐廳排序。
4. 以食材召回、保存優先級與可信內容產生食譜排序。
5. 使用 OpenCV 將使用者表情轉成可覆寫的推薦情境。
6. 建立可驗證的登入、Session 與收藏資料庫。
7. 以 React、FastAPI、Docker 建立可外部部署的完整系統。

## 三、系統架構

```text
React + Tailwind CSS
  ├─ 外食條件、推薦卡片、相機情緒
  ├─ 內食食材、推薦卡片、保存資訊
  └─ 登入、註冊、收藏
           ↓ REST API
FastAPI
  ├─ Auth / Session / Favorites
  ├─ OpenCV Emotion Pipeline
  ├─ Restaurant Recommendation Pipeline
  └─ Recipe Recommendation Pipeline
           ↓
CSV 資料集 / SQLite / ONNX 模型
```

原 Streamlit 介面保留為備援版；React/FastAPI 為目前正式展示介面。

## 四、外食推薦流程

1. 收集預算、距離、餐點類型、外帶、辣度與評分等需求。
2. 取得位置、天氣、時段及手動／OpenCV 心情。
3. 篩選餐廳候選。
4. 計算價格、距離、評分、類型、情境與 CP 值分數。
5. 分析評論情緒、負評比例與常見疑慮。
6. 依省錢、快速外帶、不踩雷等意圖調整排序。
7. 先輸出推薦清單，再提供評論與模型進階分析。

## 五、內食推薦流程

1. 輸入食材、已放天數、保存期限、價格與易腐程度。
2. 將食材別名轉成標準名稱。
3. 根據剩餘期限與浪費成本計算使用優先級。
4. 從 `recipes.csv` 召回至少命中一項食材的候選。
5. 依食材相似度、缺料、時間、熱量與保存加權進行混合排序。
6. 從 `recipe_knowledge.csv` 取得審核過的步驟與來源。
7. 先輸出推薦食譜，再提供保存排程及模型進階分析。

## 六、登入與資料庫

- 使用 SQLite 建立 `users`、`sessions`、`favorites`。
- 密碼採 `scrypt` 加鹽雜湊，不保存明文。
- 登入成功後以隨機 Token 建立七天 Session。
- Cookie 設定 `HttpOnly`、`SameSite=Lax`，線上環境啟用 `Secure`。
- 收藏資料與使用者 ID 關聯，重新登入後仍可讀取。
- Railway 必須掛載 `/app/data` Volume 才能跨部署保存 SQLite。

## 七、OpenCV 表情辨識

- YuNet：臉部偵測與關鍵點。
- MobileFaceNet：七類表情分類。
- 輸出：表情標籤、信心值、建議心情與臉部位置。
- 表情只作為推薦情境，使用者仍能手動修改。
- 照片只在記憶體處理，不寫入資料庫或檔案。

## 八、前端與使用者體驗

- React 19、Tailwind CSS 4、Vite、Lucide Icons。
- 推薦清單提前，進階模型資訊使用收合區塊。
- 外食／內食使用分段控制，避免塞入側邊欄。
- 登入和相機使用 Modal，不中斷目前條件。
- 桌面為左右工作區，手機自動改為單欄。

## 九、測試與評估

- 單元／整合測試：食材標準化、候選召回、可信內容檢索。
- API 測試：外食、內食、註冊、登入、Session、收藏與登出。
- OpenCV 錯誤測試：空白或無臉影像應回傳明確錯誤。
- 前端測試：production build、外食／內食流程、登入 Modal、相機預覽。
- 模型評估：比較評論分析、保存加權前後的排名變化。

## 十、部署

- 多階段 Docker：Node 建置 React，Python 執行 FastAPI。
- FastAPI 同時提供 `/api/*` 與 React 靜態網站。
- Railway 透過 `PORT` 啟動服務，使用 `/api/health` 健康檢查。
- SQLite 使用 `/app/data` Volume；未來可升級 PostgreSQL。

## 十一、限制與未來工作

- 目前資料集規模有限，尚未串接即時 Google Places／評論資料。
- 表情模型可能受光線、角度與遮擋影響，不作醫療用途。
- SQLite 適合單一課堂展示服務，不適合多副本高併發。
- 未來可加入 PostgreSQL、OAuth、合法地點 API、向量資料庫與有來源約束的 LLM 回答。

## 十二、結論

本專題從一般推薦網站升級為具備情境感知、影像分析、評論風險、保存排程、可信內容檢索、真實帳號資料與可部署前後端架構的智慧飲食決策系統。
