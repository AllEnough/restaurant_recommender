# 簡報大綱與講稿

建議控制在 10 至 12 分鐘，投影片以操作畫面、流程圖和結果比較為主，避免貼大量程式碼。

## 第 1 頁：封面

**標題：** 智慧飲食決策系統

**副標題：** 結合評論風險、食材保存優先級與 OpenCV 情境辨識

**畫面：** React 首頁截圖或外食／內食並列畫面。

**講稿：**

> 我們的專題不是單純隨機推薦餐廳，而是根據使用者當下情境協助做決策。外食降低踩雷風險，內食則優先處理容易過期的食材。

## 第 2 頁：研究動機

**投影片內容：**

- 外食資訊很多，但比較成本高。
- 評分無法直接代表是否符合預算與距離。
- 冰箱有食材，仍不知道能煮什麼或該先用什麼。
- 傳統推薦較少同時解釋風險、理由與資料來源。

**講稿：**

> 我們想解決的不是缺少資料，而是資料太分散，使用者仍然必須自己做最後判斷。

## 第 3 頁：專題目標

**投影片內容：**

1. 建立外食與內食兩條推薦流程。
2. 讓推薦具備情境、風險與可解釋性。
3. 優先使用快過期或浪費成本較高的食材。
4. 加入 OpenCV 表情輸入與真實帳號資料庫。
5. 完成可部署的 React/FastAPI 網站。

**畫面：** 五個目標搭配簡單圖示。

## 第 4 頁：系統架構

```text
React + Tailwind CSS
  ├─ 外食／內食操作
  ├─ 登入／收藏
  └─ 相機／定位
        ↓ REST API
FastAPI
  ├─ 推薦模型
  ├─ OpenCV
  └─ Auth + SQLite
        ↓
CSV + SQLite + ONNX
```

**畫面：** 將上方文字改成三層架構圖。

**講稿：**

> React 負責使用者操作，FastAPI 負責推薦、登入和影像模型。資料層分成推薦資料集、帳號資料庫與 ONNX 模型。正式網站由 Docker 部署在 Railway。

## 第 5 頁：外食推薦

**投影片內容：**

- 基礎權重：評分 25、價格 20、距離 15，其餘條件 40。
- 評論分析：情緒分數、負評比例、風險與常見疑慮。
- 情境意圖：省錢、快速、不踩雷。
- 環境情境：位置、天氣、時段與心情。
- 空間呈現：React Leaflet + OpenStreetMap 推薦圖針。

```text
最終分數 = 基礎分數 + 評論調整 + 情境意圖調整
```

**畫面：** 第一名推薦卡片與「評論與分數細節」展開畫面。

**地圖展示：** 說明紅色代表第一名、綠色代表高 CP，並點擊圖針展示距離、推薦分數與 Google Maps 連結。

**講稿：**

> 不同快速模式不只是改變表單預設值，還會實際加入不同的排序分數，因此省錢、快速和不踩雷的結果會有差異。

**程式展示：** 開啟 `report/algorithm_examples.py` 的 `restaurant_score()`，說明基礎分數、評論調整與情境調整如何組成最終分數。

## 第 6 頁：OpenCV 表情辨識

**流程：**

```text
相機拍照
  → YuNet 找臉與關鍵點
  → 臉部對齊
  → MobileFaceNet 分類
  → 映射推薦心情
  → 使用者確認或修改
```

**畫面：** 相機 Modal、辨識標籤與套用後的心情選單。

**講稿：**

> 表情只是一個輔助輸入，不直接代表使用者的心理狀態。照片不會保存，而且使用者可以覆寫結果。

## 第 7 頁：冰箱食材推薦

**流程：**

```text
食材標準化
  → 候選召回
  → 食譜基礎分數
  → 保存優先級
  → 混合排序
  → 可信步驟檢索
```

**分數組成：**

- 食材符合度：最高 50。
- 時間：最高 20。
- 難度：最高 20。
- 熱量：最高 10。
- 保存加權：最高 25。

**畫面：** 雞蛋、豆腐、蔥的輸入畫面與推薦結果。

## 第 8 頁：食材保存優先級

```text
優先分數 = 保存進度 + 即將到期加分 + 價格分數 + 易腐分數
排程比值 = 浪費成本 / (剩餘天數 + 1)
```

**投影片內容：**

- 剩餘天數越短，優先度越高。
- 價格越高、越容易腐敗，浪費成本越高。
- 使用高優先食材的食譜獲得額外加分。

**畫面：** 兩道食譜加入保存加權前後的排名比較。

**講稿：**

> 一般食譜推薦只問能不能煮，我們還會判斷現在應該先煮哪一道，這是內食功能最重要的差異。

**程式展示：** 開啟 `ingredient_priority()` 與 `recipe_score()`，修改豆腐的 `days_stored` 後重新執行，展示保存狀態如何改變排序。

## 第 9 頁：登入、資料庫與部署

| 資料表 | 內容 |
|---|---|
| `users` | Email、名稱、scrypt 密碼雜湊 |
| `sessions` | Token 雜湊與七天到期時間 |
| `favorites` | 使用者收藏的餐廳與食譜 |

**部署：** GitHub → Docker multi-stage build → Railway → `/app/data` Volume。

**畫面：** 登入 Modal、收藏按鈕與 Railway Active 畫面。

**講稿：**

> 登入不是前端假畫面，資料會寫入 SQLite。Railway 透過 Volume 保存資料，因此重新部署後帳號與收藏仍可保留。

## 第 11 頁：測試與限制

**測試成果：**

- Python 自動測試 6 項通過。
- React production build 通過。
- Railway 首頁、健康檢查與 Options API 回傳 HTTP 200。
- 無人臉影像會回傳明確錯誤。

**限制：**

- 專題資料集規模有限，不是即時 Google 全量評論。
- 規則式情緒分析無法完整理解反諷。
- OpenCV 可能受光線、角度與遮擋影響。
- 目前為可信檢索層，尚未完成 LLM 與向量資料庫。
