# 簡報大綱：智慧飲食決策系統

## 第 1 頁：封面

**主標題：** 智慧飲食決策系統
**副標題：** 結合評論風險、食材保存排程與 OpenCV 情境辨識

## 第 2 頁：研究動機

- 外食平台資訊很多，但使用者仍要自行比較與避雷。
- 冰箱有食材不代表知道該先煮什麼。
- 傳統推薦缺少原因、風險與可信來源。

一句話：我們要解決的不是「找資料」，而是「幫使用者做決策」。

## 第 3 頁：專題目標

1. 外食降低踩雷與搜尋成本。
2. 內食降低食材浪費。
3. 推薦結果可解釋、可追蹤。
4. 加入真實帳號、收藏與個人資料。
5. 以 OpenCV 提供自然的情境輸入。
6. 建立可部署的 React/FastAPI 完整網站。

## 第 4 頁：系統架構圖

```text
使用者
  ↓
React + Tailwind CSS
  ├─ 外食／內食推薦
  ├─ 登入／收藏
  └─ 相機／定位
  ↓ REST API
FastAPI
  ├─ OpenCV 表情辨識
  ├─ 外食推薦與評論分析
  ├─ 內食召回與混合排序
  └─ Auth + SQLite
  ↓
CSV + SQLite + ONNX
```

講解重點：React 負責體驗，FastAPI 負責推薦、帳號與模型，資料不是只放在瀏覽器裡。

## 第 5 頁：功能一－情緒與外食推薦

- 使用者輸入預算、距離、類型與用餐需求。
- 系統整合位置、天氣、時段、評論與情境意圖。
- OpenCV：YuNet 找臉、MobileFaceNet 分類表情。
- 表情映射成推薦心情，仍允許使用者手動修改。
- 推薦清單先顯示，進階分數後移。

展示：拍照 → 顯示表情與信心值 → 套用心情 → 排名改變。

## 第 6 頁：功能二－冰箱食材推薦

```text
食材輸入
  → 名稱標準化
  → 保存優先級
  → 候選召回
  → 混合式排序
  → 可信食譜檢索
  → 推薦與原因
```

- 同時考慮命中食材、缺料、時間、熱量與保存風險。
- 優先使用快過期或浪費成本較高的食材。
- 步驟取自審核資料庫，不由模型自由編造。

## 第 7 頁：功能三－真實登入與收藏

| 資料表 | 儲存內容 |
|---|---|
| `users` | Email、顯示名稱、scrypt 密碼雜湊 |
| `sessions` | Token 雜湊、到期時間、使用者 ID |
| `favorites` | 每位使用者的餐廳與食譜收藏 |

展示：註冊 → 收藏 → 登出 → 重新登入 → 收藏仍存在。

## 第 8 頁：開發工具與技術

| 層級 | 技術 |

| 前端 | React 19、Tailwind CSS 4、Vite、Lucide |
| 後端 | Python、FastAPI、Pydantic、Uvicorn |

## 第 9 頁：系統成果展示

建議依此順序現場操作：

1. 開啟 React 首頁，說明推薦結果優先的版面。
2. 外食套用「大學生省錢午餐」並送出。
3. 展開第一名的評論與分數細節。
4. 開啟相機辨識，套用心情後重新推薦。
5. 註冊帳號並收藏第一名。
6. 切換內食，輸入雞蛋、豆腐、蔥與保存天數。
7. 展示保存加權、缺少食材與可信料理步驟。
8. 登出再登入，確認收藏仍存在。

畫面重點：不要先展示 Dashboard；先讓老師看到系統真的能完成決策。

## 第 10 頁：測試與成果

- 推薦、登入、Session、收藏、OpenCV API 自動測試通過。
- React production build 通過。
- 外食與內食實際操作皆能產生五筆推薦。
- 無臉影像會回傳可理解錯誤，不會誤當辨識成功。
- 密碼不以明文存入資料庫。

## 第 11 頁：限制與未來發展

- 擴充真實餐廳與評論資料來源。
- SQLite 升級 PostgreSQL，加入 Migration 與 OAuth。
- 擴充食譜知識庫與向量檢索。
- 串接 LLM，但只允許依檢索來源生成回答。
- 進行更多使用者測試與推薦指標評估。

## 第 12 頁：結論

智慧飲食決策系統將推薦演算法、評論風險、保存排程、可信檢索、OpenCV、登入資料庫與前後端部署整合在同一套產品原型中，讓使用者更快決定外食選擇，也更有效利用冰箱食材。
