# 智慧飲食決策系統專案說明

## 一、專案定位

智慧飲食決策系統不是單純隨機抽餐廳，而是面向兩種族群的決策輔助平台：外食族降低搜尋與踩雷成本；內食族根據冰箱狀態決定應該先使用哪些食材。系統將使用者條件、評論風險、環境情境、食材保存期限與可信食譜內容轉換成可解釋的推薦排序。

## 二、主要成果

1. 推薦結果優先：送出條件後先看到餐廳或食譜清單，進階分析置於後段。
2. 前後端分離：React/Tailwind 負責操作介面，FastAPI 提供推薦、帳號及影像分析 API。
3. 真實帳號資料：SQLite 儲存帳號、Session 與收藏；密碼以 `scrypt` 加鹽雜湊。
4. 表情情境輸入：OpenCV 偵測人臉並分類七種表情，再映射到既有心情推薦策略。
5. 外食混合決策：整合價格、距離、評分、天氣、時段、評論與快速情境。
6. 內食混合決策：整合食材召回、料理限制、保存優先級與審核食譜內容。
7. 可部署架構：Docker 同時封裝 React production build、FastAPI、ONNX 模型及資料集。
8. 空間化結果：React Leaflet 將推薦餐廳、排名、CP 值與使用者位置呈現在 OpenStreetMap。
9. Persona 情境：大學生、上班族、老師、宅家與健身模式會套用不同條件，而不是只變更標籤。

## 三、系統流程

### 外食

```text
使用者條件／相機表情／位置
  → 條件正規化
  → 餐廳候選篩選
  → 基礎條件加權
  → 評論風險與情境意圖調整
  → 推薦排序與理由
  → 推薦清單
  → 進階分數分析
```

### 內食

```text
使用者輸入食材與保存狀態
  → 食材名稱標準化與分級
  → 計算食材使用優先級
  → 從食譜資料集召回候選
  → 混合式推薦排序
  → 檢索審核食譜內容
  → 推薦清單、料理步驟與推薦原因
  → 進階保存排程分析
```

## 四、前端設計

React 介面採用工作型應用布局，不使用大型行銷頁：

- 外食／內食使用分段控制切換。
- 左側集中輸入條件，右側立即呈現推薦答案。
- 推薦卡片顯示分數、理由、風險與收藏按鈕。
- 評論、模型拆分、料理步驟與保存資訊使用展開區塊。
- 登入及相機辨識使用 Modal，不離開當前推薦情境。
- 手機畫面改為單欄，避免文字與控制項重疊。
- 進階分析在手機版限制於卡片寬度，寬表格採局部水平捲動，避免整頁橫向破版。
- 外食清單後方提供互動地圖，圖針以第一名、高 CP、中 CP、一般推薦與使用者位置分色。
- 外食側欄顯示自動天氣、自動時段與距離來源，並保留手動覆寫。
- 內食食材可編輯價格及易腐程度，直接影響保存優先級與混合排序。
- 登入後頁首收藏數量可點擊，開啟收藏清單並移除項目。
- 食譜可信來源區顯示食材、內容編號、審核日期、產地與有效期限資訊。

## 五、後端與資料庫

FastAPI 提供以下 API：

| 路徑 | 功能 |
|---|---|
| `GET /api/health` | 服務健康檢查 |
| `GET /api/options` | 取得可用選項 |
| `POST /api/recommend/restaurants` | 外食推薦 |
| `POST /api/recommend/recipes` | 內食推薦 |
| `POST /api/emotion/analyze` | OpenCV 表情辨識 |
| `POST /api/auth/register` | 註冊並建立 Session |
| `POST /api/auth/login` | 登入 |
| `POST /api/auth/logout` | 登出並刪除 Session |
| `GET /api/auth/me` | 取得目前登入者 |
| `GET/POST/DELETE /api/favorites` | 收藏管理 |

SQLite 資料表：

| 資料表 | 用途 |
|---|---|
| `users` | 帳號、顯示名稱、密碼雜湊、建立時間 |
| `sessions` | Token 雜湊、使用者、到期時間 |
| `favorites` | 使用者收藏的餐廳與食譜 |

此設計與 Django 的 Model/Session 概念相近，但目前沒有使用 Django ORM、Admin 或 Migration；資料表由 FastAPI 啟動時以 SQL 建立。對課堂展示而言是真實可驗證的後端，不是前端假登入。

## 六、OpenCV 表情辨識

處理管線：

1. 瀏覽器取得相機畫面。
2. 使用者主動拍照後，上傳單張 JPEG 至 FastAPI。
3. YuNet 偵測臉部與五個關鍵點。
4. 對齊臉部、轉為 64 × 64 灰階影像，再交由 FER+ CNN 分類。
5. 將 neutral、happiness、surprise、sadness、anger、disgust、fear、contempt 映射成推薦心情。
6. 使用者確認或手動修改後再進行推薦。

系統不保存影像，並明確標示這不是心理或醫療評估。

## 七、技術含量

- 多來源特徵整合與可解釋加權推薦。
- 三個可獨立執行且由正式 API 直接匯入的核心：`review_score.py`、`recipe_rank.py`、`ingredient.py`。
- 評論文字規則分析、負評風險與排名調整。
- Haversine 空間距離與瀏覽器定位。
- 食材標準化、集合召回與混合排序。
- 保存期限／浪費成本導向的排程加權。
- 具有九欄結構（食材、食譜、內容編號、步驟、提醒、來源、審核日、產地、有效期限）的可信食譜檢索層；缺少必要來源欄位的候選不會進入最終結果。
- OpenCV DNN、YuNet 人臉偵測與 ONNX FER+ 八分類。
- REST API、Cookie Session、密碼雜湊與資料庫關聯。
- React 狀態管理、相機 API、登入流程與響應式介面。
- Docker 多階段建置與 Railway 持久化 Volume。

### 進階分析與模型資訊

React 正式版已恢復原 Streamlit 版的核心分析能力，並改由 FastAPI 回傳結構化分析資料：

- 外食決策 Dashboard：候選數、平均最終分、平均負評與低風險餐廳數。
- 外食模型評估：基礎排序與評論／情境增強排序的重疊率、排名變化與負評差異。
- 外食分數拆解：基礎分、評論調整、情境調整與最終分數。
- 權重敏感度：比較綜合、省錢、距離、評分、口碑及低負評策略的第一名。
- 內食決策 Dashboard：候選數、可直接料理數、平均缺料與平均保存加權。
- 內食模型評估：基礎排序與保存優先排序的重疊率、排名變化及高優先食材命中。
- 食材標準化、保存優先表、可信內容覆蓋與食譜各項分數拆解。

推薦清單仍位於前方，完整進階區塊放在清單後方並預設展開，也可透過按鈕收合。若頁面仍保留部署前的舊推薦資料，系統會顯示重新計算提示，不會呈現空白區塊。

## 八、部署與限制

正式版使用一個 Docker 容器：Node 階段建置 React，Python 階段執行 FastAPI，FastAPI 再提供前端靜態檔。Railway 部署時需將 Volume 掛載至 `/app/data`，並設定 `APP_DB_PATH=/app/data/app.db`，才能保存 SQLite 帳號。

目前限制包括資料集規模有限、評論並非即時 Google 全量資料，以及 SQLite 不適合多副本服務。後續若擴充為正式產品，可改用 PostgreSQL、OAuth、資料遷移工具與合法授權的地點／評論 API。
