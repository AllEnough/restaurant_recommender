# 智慧飲食決策系統專案說明

## 一、專案簡介

本專題名稱為「智慧飲食決策系統」，是一套使用 Python 與 Streamlit 開發的互動式網站系統。系統主要面向兩種使用者情境：外食族與內食族。

外食族常需要在短時間內比較餐廳的價格、距離、評分、餐點類型與評論風險；內食族則常遇到冰箱有食材，但不知道可以煮什麼，或食材快過期卻沒有優先使用的問題。

因此，本系統希望不只是提供「今天吃什麼」的簡單推薦，而是建立一套能根據使用者條件、評論資料、食材狀態與模型評估結果進行判斷的飲食決策輔助工具。

## 二、專案目標

本專題的主要目標如下：

1. 建立外食餐廳與內食食譜兩種資料集。
2. 使用 Python 與 pandas 進行資料讀取、整理與推薦計算。
3. 設計外食餐廳多條件加權推薦模型。
4. 加入餐廳評論文字分析，計算情緒分數、負評比例與常見關鍵字。
5. 加入情緒策略，讓使用者心情實際影響餐廳推薦排序。
6. 加入自動天氣判斷，將即時天氣轉換成推薦模型條件。
7. 加入自動用餐時段判斷，依照早餐、午餐、下午茶、晚餐與宵夜調整推薦。
8. 加入瀏覽器定位，依照使用者實際位置重新計算餐廳步行距離。
9. 使用 folium 建立互動式餐廳地圖。
10. 設計內食食譜食材比對模型。
11. 加入食材保存優先級排序，優先使用快過期或浪費成本較高的食材。
12. 提供決策分析 Dashboard，整理推薦結果的重點數據。
13. 提供模型評估，比較模型調整前後的推薦排名差異。
14. 提供推薦模型解釋器，拆解外食推薦分數來源。
15. 使用 Streamlit 建立可互動、可部署的網站介面。

## 三、使用者情境

### 1. 外食決策

外食使用者可以輸入預算、距離、餐點類型、天氣來源、用餐時段來源、位置來源、心情、外帶需求、辣度與出餐速度偏好。系統可自動取得所在地天氣，也可依照目前時間自動判斷用餐時段；若使用者允許瀏覽器定位，系統會依照使用者實際位置重新計算餐廳步行距離。若使用者想自行控制條件，也可以改成手動選擇。系統會依照這些條件計算餐廳推薦分數，並結合評論分析結果，協助使用者降低踩雷風險。

外食模式適合以下情境：

- 不知道今天要吃什麼。
- 想快速找到符合預算與距離的餐廳。
- 想避開評論風險較高的餐廳。
- 想根據心情選擇適合的餐點類型。
- 想讓系統依照天氣與用餐時段自動調整推薦。
- 想讓系統依照目前所在位置重新計算餐廳距離。
- 想在地圖上查看推薦餐廳位置。

### 2. 內食決策

內食使用者可以輸入冰箱現有食材，並設定料理時間、難度、熱量上限與可接受缺少食材數。系統會比對食譜資料，推薦最適合目前食材狀態的料理。

內食模式適合以下情境：

- 冰箱有食材，但不知道能煮什麼。
- 想減少食材浪費。
- 想優先使用快過期或價格較高的食材。
- 想知道還缺哪些食材。
- 想快速找出料理時間較短的食譜。

## 四、系統功能

### 1. 外食餐廳推薦

系統根據以下條件推薦餐廳：

- 預算上限
- 可接受距離
- 餐點類型
- 目前天氣，自動偵測或手動選擇
- 用餐時段，自動判斷或手動選擇
- 目前位置，瀏覽器定位或手動據點
- 目前心情
- 是否需要外帶
- 可接受辣度
- 是否希望快速出餐
- 排序方式
- 最低評分

輸出內容包含：

- 推薦餐廳排名
- 推薦分數
- 推薦理由
- 推薦標籤
- 評論摘要
- 自動天氣判斷結果
- 自動用餐時段判斷結果
- 使用者定位與重新計算後的步行距離
- 餐廳地圖
- 完整餐廳資料表

### 2. 情緒策略推薦

外食模式支援自動天氣判斷。使用者輸入所在地區後，系統會透過公開天氣資料取得目前溫度與天氣描述，並轉換成推薦模型使用的四種分類：普通、熱、冷、雨天。若天氣資料取得失敗，系統會自動使用普通天氣作為備援，也可以改成手動選擇。

系統也會依照目前時間自動判斷用餐時段，例如早餐、午餐、下午茶、晚餐與宵夜，並提高對應餐點類型的推薦分數。使用者也可以改為手動選擇時段。

系統也支援瀏覽器定位。使用者允許定位後，系統會取得目前經緯度，使用 Haversine 公式計算使用者到每間餐廳的距離，並換算成預估步行分鐘。若使用者不允許定位，系統也提供逢甲大學、逢甲夜市、文華路商圈等手動據點作為備援。

用餐時段與推薦類型對應如下：

| 用餐時段 | 加權餐點類型 |
|---|---|
| 早餐 | 早餐、早午餐、飯類、飲料 |
| 午餐 | 便當、飯類、麵食、小吃、水餃、鍋貼、素食 |
| 下午茶 | 飲料、甜點、輕食、早午餐 |
| 晚餐 | 火鍋、麵食、飯類、日式、韓式、義式、泰式、鐵板燒、美式、港式 |
| 宵夜 | 小吃、炸物、飲料、麵食、鍋貼、水餃 |

系統會根據使用者選擇的心情調整推薦策略。

| 心情 | 推薦策略 |
|---|---|
| 省錢 | 低價與 CP 值高的餐廳加分 |
| 疲累 | 距離近與快速出餐的餐廳加分 |
| 開心 | 高評分與適合聚餐的餐廳加分 |
| 心情不好 | 甜點、飲料、炸物、小吃加分 |
| 選擇困難 | 從高分餐廳中產生驚喜推薦 |

### 3. 餐廳評論文字分析

系統使用 `reviews.csv` 模擬餐廳評論資料，並透過關鍵字方式進行情緒分析。

分析內容包含：

- 評論數量
- 評論情緒分數
- 負評比例
- 常見優點
- 常見疑慮
- 常見主題
- 評論摘要
- 評論調整分數

評論調整分數會影響外食推薦的最終排序，使推薦結果不只依賴價格、距離與評分，也能考慮使用者評論中的風險。

### 4. 虛擬美食地圖

系統使用 folium 與 streamlit-folium 製作互動式地圖，將推薦餐廳標示在地圖上。

圖針顏色意義如下：

| 顏色 | 意義 |
|---|---|
| 紅色 | 本次推薦第一名 |
| 綠色 | 高 CP 值餐廳 |
| 橘色 | 中高 CP 值餐廳 |
| 藍色 | 一般推薦餐廳 |

使用者點擊圖針後，可以查看餐廳名稱、餐點類型、價格、評分、距離、推薦分數、CP 值與推薦理由。

### 5. 內食食譜推薦

使用者輸入冰箱現有食材後，系統會比對 `recipes.csv` 中的食譜資料，計算每道食譜的符合程度。

推薦條件包含：

- 現有食材
- 可接受烹飪時間
- 料理難度
- 熱量上限
- 最多可缺少食材數
- 是否只顯示現有食材足夠的食譜

輸出內容包含：

- 食譜名稱
- 食譜類型
- 料理時間
- 難度
- 熱量
- 符合食材
- 缺少食材
- 推薦分數
- 食材優先加權
- 推薦理由
- 缺少食材購物清單

### 6. 食材保存優先級

內食模式中，使用者可以為每個食材設定：

- 已放天數
- 保存期限
- 估計價格
- 易腐程度

系統會計算：

- 優先分數
- 剩餘天數
- 排序比值
- 優先級高、中、低

此功能參考鞋匠排程問題的概念：如果一個食材越快過期、價格越高、越容易腐壞，代表浪費成本越高，因此應該優先被使用。系統會將這個優先級轉換成食譜推薦加權，讓內食推薦不只是判斷「能不能煮」，也會判斷「應該先煮什麼」。

### 7. 決策分析 Dashboard

系統提供外食與內食兩種 Dashboard。

外食 Dashboard 顯示：

- 決策候選數
- 最佳選項
- 平均負評比例
- 評論低風險餐廳數
- 原始分數與最終分數比較
- 不同餐點類型的平均負評比例與評論情緒

內食 Dashboard 顯示：

- 目前食材數
- 可直接料理數
- 平均缺少食材數
- 高優先食材估計價值
- 食譜符合食材數
- 缺少食材數
- 料理時間
- 保存加權
- 不同食譜類型的平均時間與平均熱量

### 8. 模型評估

模型評估用來比較系統加入進階模型前後的推薦差異。

外食模型評估比較：

- 未納入評論分析的推薦結果
- 納入評論分析後的推薦結果
- Top 推薦重疊率
- 第一名是否改變
- 平均負評比例變化
- 平均評論情緒變化
- 餐廳原始排名與評論後排名

內食模型評估比較：

- 未納入食材保存優先級的推薦結果
- 納入食材保存優先級後的推薦結果
- Top 推薦重疊率
- 第一名是否改變
- 高優先食材使用次數
- 平均缺少食材變化
- 食譜原始排名與保存加權後排名

此功能可用來說明模型調整是否真的影響推薦結果，讓專題更接近資料分析與決策系統，而不是單純的清單推薦。

### 9. 推薦模型解釋器

外食模式提供模型解釋器，可以選擇任一推薦餐廳並查看分數來源。

分數來源包含：

- 評分貢獻
- 價格貢獻
- 距離貢獻
- 餐點類型貢獻
- 天氣與心情情境貢獻
- 外帶貢獻
- 辣度貢獻
- 出餐速度貢獻
- 心情策略加權
- 評論調整分數

此功能讓推薦結果具備可解釋性，使用者可以理解為什麼某間餐廳會被推薦。

## 九、核心模型設計

### 1. 外食推薦分數

```text
餐廳推薦分數 =
評分分數
+ 價格符合度
+ 距離符合度
+ 類型符合度
+ 天氣與心情符合度
+ 外帶符合度
+ 辣度符合度
+ 出餐速度分數
+ 情緒策略加權
+ 用餐時段加權
```

若啟用評論分析，系統會再加入評論調整分數：

```text
外食最終分數 =
餐廳推薦分數
+ 評論調整分數
```

### 2. 內食推薦分數

```text
食譜基礎分數 =
食材符合度
+ 烹飪時間符合度
+ 難度符合度
+ 熱量符合度
- 缺少食材懲罰
```

若食譜使用到高優先級食材，系統會加入保存優先級加權：

```text
食譜最終分數 =
食譜基礎分數
+ 食材保存優先級加權
```

### 3. 評論分析分數

評論分析使用正向與負向關鍵字進行情緒判斷。

系統會計算：

- 正向關鍵字命中數
- 負向關鍵字命中數
- 情緒分數
- 負評比例
- 評論風險等級
- 評論調整分數

### 4. 食材保存優先級

食材保存優先級會綜合以下因素：

- 已放天數
- 保存期限
- 剩餘天數
- 估計價格
- 易腐程度

當食材越接近過期、價格越高、越容易腐壞時，優先分數會越高。

## 十、技術含量說明

本專題的技術含量主要體現在以下幾點：

1. 多條件加權推薦模型：將使用者需求轉換成可計算分數。
2. 情緒策略加權：不同心情會改變推薦分數，而不只是單純篩選。
3. 自動天氣判斷：將即時天氣轉換成普通、熱、冷、雨天，作為推薦模型情境條件。
4. 自動用餐時段判斷：依照目前時間推論早餐、午餐、下午茶、晚餐或宵夜，作為推薦加權條件。
5. 使用者位置感知推薦：透過瀏覽器定位取得經緯度，重新計算餐廳距離並影響距離分數。
6. 評論文字分析：將文字評論轉換成情緒分數、負評比例與風險等級。
7. 可解釋推薦：提供分數來源拆解，讓推薦結果透明化。
8. 模型評估：比較模型調整前後的排名差異與平均風險變化。
9. 食材集合比對：計算現有食材與食譜所需食材的符合程度。
10. 食材保存優先級：參考排程問題概念，優先使用浪費成本較高的食材。
11. 決策分析 Dashboard：將推薦結果轉換成指標、表格與摘要。
12. 互動式地圖：用地圖與圖針顏色呈現空間資訊與 CP 值差異。
13. Streamlit 介面整合：將資料、模型、分析與視覺化整合成可操作網站。

## 十二、預期展示流程

建議展示時可以依照以下順序：

1. 介紹系統首頁與專題目標。
2. 切換到外食決策模式。
3. 輸入預算、距離、心情與餐點類型。
4. 展示外食推薦結果與推薦理由。
5. 展示評論分析，說明情緒分數與負評比例。
6. 展示外食 Dashboard，說明推薦結果的整體風險。
7. 展示外食模型評估，比較評論分析前後的推薦差異。
8. 展示推薦模型解釋器，拆解餐廳分數來源。
9. 展示互動式餐廳地圖。
10. 切換到內食決策模式。
11. 輸入冰箱現有食材。
12. 設定食材保存資訊。
13. 展示內食推薦結果與缺少食材購物清單。
14. 展示內食 Dashboard 與模型評估。
15. 說明食材保存優先級如何降低食材浪費。

## 十三、目前限制與未來改善

目前限制：

- 餐廳、評論與食譜資料為 CSV 資料集，規模有限。
- 評論分析使用關鍵字方式，尚未使用大型語言模型或深度學習模型。
- 餐廳評論為模擬資料，尚未串接真實 Google Maps 評論。
- 使用者偏好目前不會長期儲存。
- 食材保存資訊需要使用者手動輸入。

未來改善方向：

- 串接真實餐廳 API 或地圖資料。
- 加入更完整的自然語言處理模型進行評論摘要。
- 加入使用者偏好紀錄，讓推薦結果越用越準。
- 加入登入系統與個人收藏資料庫。
- 加入食材條碼或影像辨識，自動建立冰箱食材清單。
- 擴充資料集，提高推薦結果的真實性與泛用性。

## 十四、專題總結

本專題從「餐廳與食譜推薦」出發，進一步升級成「智慧飲食決策系統」。系統同時支援外食與內食兩種情境，並結合推薦模型、評論文字分析、食材保存優先級、互動式地圖、Dashboard 與模型評估。

相較於單純的推薦清單，本系統更強調決策輔助與可解釋性。使用者不只知道系統推薦什麼，也能理解為什麼推薦、模型調整前後有什麼差異，以及如何根據資料降低外食踩雷風險與內食食材浪費。
